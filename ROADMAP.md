# Roadmap

## Приоритетные задачи

### 1. Сделать разбиение на чанки естественным и предсказуемым
- Сохранять целостность абзацев, списков, таблиц, цитат и fenced code blocks, когда это возможно в пределах лимита VK.
- Выбирать границы чанков так, чтобы текст оставался читаемым и не выглядел случайно разрезанным.
- Если разрез внутри блока неизбежен, делать его последовательно и с минимальной потерей смысла.
- Гарантировать корректность `items` на границах чанков для длинных форматированных фрагментов.
- Закрепить ожидаемое поведение тестами на длинные и смешанные документы.

### 2. Стабилизировать поддерживаемый subset Markdown
- Сфокусироваться на надёжности уже поддерживаемых конструкций, а не на хаотичном расширении синтаксиса.
- Снизить количество surprising cases, где результат выглядит технически валидным, но не совпадает с ожиданием автора markdown.
- Привести в порядок пограничные случаи для inline-форматирования, заголовков, таблиц, цитат, экранирования и code fences.
- Явно отделить гарантированно поддерживаемое поведение от конструкций, которые намеренно остаются литеральным текстом.
- Держать README и тесты синхронизированными с фактическим поведением библиотеки.

### 3. Довести таблицы до стабильного визуального качества
- Улучшить выравнивание колонок для смешанных языков, цифр, пунктуации, emoji и форматированного текста.
- Снизить число случаев, когда таблица выглядит ровной только на части строк, а на остальных заметно “плывёт”.
- Сохранить предсказуемость авто-выравнивания для числовых и текстовых колонок.
- Проверять качество таблиц не только функционально, но и по эталонным визуальным сценариям.
- Сделать поведение таблиц достаточно стабильным для типовых сообщений, которые реально отправляют в VK.

### 4. Упростить интеграцию библиотеки с VK API
- Снизить объём повторяющегося glue-кода в местах, где результат `render` сразу превращается в payload для отправки.
- Сделать типовой сценарий использования понятным без ручной сборки `format_data` в каждом проекте.
- Сохранить низкоуровневый API для интеграций, которым нужен полный контроль над чанками и `items`.
- Сделать интеграционный сценарий более очевидным в README и публичном API.

## Кандидаты после стабилизации основного поведения

### 5. Точечно улучшить производительность на реальных сценариях
- Опираться на измеримые кейсы, а не на предположения о будущих узких местах.
- Отдельно оценивать длинные plain-text сообщения, тяжёлые inline-кейсы, крупные таблицы и повторные рендеры.
- Не усложнять код ради ускорений, которые не дают заметного выигрыша в реальном использовании.
- Рассматривать оптимизации только там, где они не ухудшают читаемость и сопровождение библиотеки.

### 6. Расширять синтаксис только там, где это заметно повышает ценность библиотеки
- Добавлять новые markdown-конструкции только при понятной пользе для сообщений VK, а не ради формальной полноты.
- Проверять, что новая поддержка действительно улучшает пользовательский результат, а не просто усложняет парсер.
- Для каждого нового синтаксического блока сразу фиксировать ожидаемое поведение в README и тестах.
- Не размывать фокус библиотеки: приоритет у качества конвертации, а не у гонки за full Markdown compatibility.
